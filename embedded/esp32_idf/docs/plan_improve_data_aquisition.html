Summary

The acquisition architecture is moving from synchronous ADC oneshot reads toward a producer/consumer model. The signal loop must stay deterministic: it should read an already-published analog snapshot, compute the next correction, and apply that correction on the next cycle boundary. It must not call ADC drivers, allocate memory, encode protobuf, send BLE data, or wait for a sample.

The first observability milestone is complete: analog snapshots, status counters, diagnostic results, dashboard rendering, and diagnostic commands exist. The current DMA milestone is partially implemented: the firmware can switch between rate-limited oneshot and ADC continuous/DMA mode with dashboard commands. The remaining work is to harden the DMA path from "experimental producer" into the control-grade acquisition engine.

The recommended path remains: keep rate-limited oneshot as the safe dashboard monitor mode, finish ADC continuous/DMA as the real acquisition engine, then accept it only if oscilloscope timing proves the 2 us dead-time is not degraded.

Decisions Defined So Far

Timing Control cycle target A fresh calibrated AN3/AN5/AN6 triple, control math, and command generation must fit inside the roughly 280 us signal cycle.
Rate First target rate Target 4 complete triples per 280 us cycle, or one complete triple about every 70 us.
Freshness Consumption rule The signal/control path consumes the latest complete snapshot. It never blocks waiting for ADC.
Fault Miss policy If the sample is stale, reuse the previous correction, count a miss, and disable closed-loop correction after 3 consecutive misses.
Calibration Hot path Replace the current per-sample linear calibration scan with a raw-code LUT or fixed-point interpolation in internal RAM.
Scope gate Final authority No diagnostic counter can override the oscilloscope. The 2 us dead-time must stay stable during the DMA diagnostic run.
Decision Implementation consequence
AN3, AN5, and AN6 do not need true simultaneous sampling for the first version. DMA frames can be parsed into fast sequential triples. Publish only when all three channels are present.
BLE telemetry is on-demand and lower priority than signal, acquisition, and control. Periodic telemetry must be disabled or throttled while control is active. BLE replies may lag under load.
Acquisition starts at boot after ADC initialization. Snapshots and counters are warm before the user enables control or runs diagnostics.
Do not silently fall back to a slow path for closed-loop control. If DMA cannot meet the target, keep open-loop signal generation available and report a structured fault.

Current State

The firmware now exposes an acquisition mode switch. The default is safe oneshot acquisition; continuous ADC is explicit and command-driven.

Area Implemented Remaining risk
Oneshot fallback ANALOG_ACQ_MODE_ONESHOT remains the boot default. The spacing is controlled by analog.set_acquisition_period. This mode is useful for dashboard/status, but it is not expected to satisfy the 280 us control freshness target.
ADC continuous/DMA analog.set_acquisition_mode {"mode":"continuous"} starts the continuous path. analog.set_continuous_sample_rate changes sample rate. The parser currently publishes triples opportunistically; it still needs stronger frame accounting, overflow policy, and hardware validation.
Snapshot/status Complete triples publish through the existing analog snapshot/status surface and still feed compatibility globals. The publication boundary should be hardened so mixed-channel reads are provably impossible for control consumers.
Diagnostic test debug.analog_test_run and debug.analog_test_result provide run summaries. The diagnostic must be extended from "system observation" to an explicit acceptance gate for DMA and control enable.

Detailed Plan to Conclude DMA Implementation

1

Extract analog acquisition into a module boundary

Move ADC continuous handle ownership, DMA frame parsing, sample-rate configuration, snapshot publishing, and counters out of main.cpp into the analog acquisition module. Keep main.cpp responsible for task creation only.

Verification: firmware build passes, mode switching still works, and system.get_status reports analog snapshots after boot.

2

Make snapshot publication atomic at the triple level

Replace independent atomic fields with a seqlock-style snapshot: producer increments an odd sequence while writing, writes raw/calibrated values and timestamp, then increments to an even sequence. Consumers retry if the sequence changes or is odd.

Verification: a stress diagnostic can read snapshots repeatedly without ever observing a partially updated AN3/AN5/AN6 triple.

3

Harden DMA frame parsing

Add explicit per-frame accounting: samples read, samples rejected, channel order anomalies, partial triples, frame drops, pool flushes, and DMA overflow count. On overflow, invalidate the current partial triple, keep the last valid snapshot visible, and let the miss policy decide whether control remains enabled.

Verification: intentionally overload the sample rate and confirm overflows increase without publishing invalid triples or blocking the signal loop.

4

Move calibration to a hot LUT

Build a 4096-entry raw-code-to-calibrated-float lookup table in internal RAM during analog initialization. The DMA producer then converts raw samples with one indexed load instead of scanning the calibration breakpoints for every sample.

Verification: compare LUT output against the current calibration function across all raw codes; error must stay within one ADC LSB equivalent or 5 mV, whichever is larger.

5

Wire control consumption to snapshots

Update the signal/control path to read only the analog snapshot API. Remove direct dependence on g_adc_an3, g_adc_an5, g_adc_an6, and g_adc_fresh after the snapshot API is proven.

Verification: no ADC driver calls, BLE sends, calibration scans, logging, allocations, or waits exist inside the interrupt-disabled signal playback block.

6

Add control-grade fault gates

Add structured reasons for stale sample, missing triple, DMA overflow, calibration unavailable, and control math late. Closed-loop control is disabled after 3 consecutive misses, but open-loop signal generation continues.

Verification: induced stale samples disable closed-loop corrections and the dashboard reports the exact fault reason.

7

Run the 60 second acceptance diagnostic

Run continuous ADC at the target rate while the diagnostic signal is active. Record analog counters, signal timing counters, BLE behavior, and oscilloscope dead-time.

Verification: 60 seconds at target rate, zero signal timing regression, zero unhandled overflow, zero 3-miss trips, and scoped 2 us dead-time remains stable.

Expected Timing Result

The expected final behavior is that ADC conversion pacing moves into hardware/DMA while Core 0 parses short frames and publishes snapshots. Core 1 should continue signal playback without observable widening or jitter in the 2 us dead-time window.

The 70 us triple spacing is the target derived from 4 triples per 280 us cycle. The current default command value of 12000 samples/s is conservative for bring-up; final target testing likely needs about 42900 channel samples/s or higher, depending on ESP-IDF sampling overhead and accepted margin.
