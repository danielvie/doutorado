Signal Timing Post Mortem

This post mortem explains why the generated waveform had an extra 2 to 2.6 us between the last segment of one pattern and the first segment of the next pattern, and why the current fix removes that extra time from normal playback.

SymptomThe last visible signal state was longer than requested by about 2 us.
ImpactThe next waveform did not start exactly after the configured segment time.
CauseLoop maintenance ran between playback batches while the last GPIO state stayed active.
FixCarry the next absolute edge deadline across loop iterations.

Expected Behavior

The requested behavior is simple: if the last segment duration is Tlast, the next wave must begin exactly after Tlast. Runtime work such as telemetry, analog-read triggering, dataset checks, and timing snapshots must not silently add time to that last state.

What Went Wrong

The original playback helper was locally accurate inside a batch, but the batch boundary was not continuous. After the final segment finished, the function waited until the final deadline, returned to signal_loop_task, and then maintenance code executed before the next call to execute_signal_pattern(). During all of that time, the final GPIO state remained active.

Why The First Change Was Not Enough

The first change moved the per-segment wait before the GPIO write, which fixed compute overhead between segments inside execute_signal_pattern(). However, it still left the batch boundary exposed: the code after execute_signal_pattern() still ran before the next batch wrote its first GPIO state.

Code Location: The Loop And Its Maintenance

Current location: main/src/signal_controller.cpp, inside signal_loop_task() around lines 561-581.

These statements are the maintenance work that took time between playback batches. Before the final fix, that work happened after the last segment had already consumed its requested time, so the final GPIO state stayed active for this extra work. 

while (signal_is_running()) {
    uint32_t loop_start = esp_cpu_get_cycle_count();
    apply_pending_dataset_swap(ctx);       // pre-playback maintenance
    update_control_corrections(ctx);     // pre-playback maintenance

    SignalTimingSample timing;
    uint32_t playback_start = esp_cpu_get_cycle_count();
    const uint32_t nominal_cycles = get_nominal_pattern_cycles(ctx);
    const uint32_t repeat_count =
        nominal_cycles < MIN_MAINTENANCE_PERIOD_CYCLES
            ? SHORT_PATTERN_BATCH_CYCLES
            : 1;

    execute_signal_pattern(ctx, timing, repeat_count);

    // post-playback maintenance that caused the visible inter-wave stretch
    uint32_t playback_cycles =
        (esp_cpu_get_cycle_count() - playback_start) / repeat_count;
    if (timing.scheduled_period_cycles >= MIN_MAINTENANCE_PERIOD_CYCLES) {
        trigger_periodic_analog_read();
    } else {
        timing.maintenance_skipped_count++;
    }
    uint32_t loop_cycles = esp_cpu_get_cycle_count() - loop_start;
    update_signal_timing_snapshot(playback_cycles, loop_cycles, timing);
}

Code Location: The Boundary Fix

Current location: main/src/signal_controller.cpp, inside execute_signal_pattern() around lines 450-512.

The fix is that the playback function starts from the saved absolute edge and saves the next edge before returning. There is no final wait after the last GPIO write, so the loop maintenance now runs during the valid hold time of the final segment. 

uint32_t pattern_start = ctx.next_pattern_edge;
if (pattern_start == 0) {
    pattern_start = esp_cpu_get_cycle_count();
}
uint32_t next_edge = pattern_start;

// ... for each segment: wait until next_edge, write GPIO, then advance next_edge
wait_until_cycle(next_edge);
GPIO.out_w1tc = step.clr_mask;
GPIO.out_w1ts = step.set_mask;
next_edge += cycles;

// No wait here. The next loop iteration will honor this deadline.
ctx.next_pattern_edge = next_edge;

Final Fix

The final fix makes the batch boundary part of the same absolute schedule. SignalLoopContext now stores next_pattern_edge. The playback helper starts from that saved deadline when it exists, and it saves the next deadline before returning. The key point is that the helper no longer waits at the end of the batch. It returns immediately after writing the final segment, while that final segment is supposed to be held. 

struct SignalLoopContext {
    const DataSet *dataset = &g_dataset_a;
    uint32_t next_pattern_edge = 0;
    int32_t current_correction[MAX_SIGNAL_SIZE];
    float dtk_buffer[MAX_SIGNAL_SIZE];
};

uint32_t pattern_start = ctx.next_pattern_edge;
if (pattern_start == 0) {
    pattern_start = esp_cpu_get_cycle_count();
}

/* write each segment at its absolute edge */

ctx.next_pattern_edge = next_edge;

Control Flow Diagram

The fixed design treats playback as one continuous timeline, even though the FreeRTOS task still calls the helper repeatedly. The loop can do maintenance between calls, but the next call is anchored to the saved edge.

Why This Fix Works

ProblemFix behaviorResult
Maintenance ran after the final wait and before the next first write. The final wait is no longer performed at the end of the batch. Maintenance happens during valid final-state hold time.
Each helper call previously started from its own current cycle count. The next helper call starts from ctx.next_pattern_edge. Batch boundaries follow one continuous absolute timeline.
Late execution looked like a silent waveform stretch. The helper checks now - next_edge before each write. Late starts are visible through overrun_count.
New datasets could inherit an old absolute deadline. apply_pending_dataset_swap() resets next_pattern_edge. Pattern swaps start from a clean local timeline.

How To Validate

  • Measure the boundary between the last segment and the first segment of the next wave on the scope.
  • The measured interval should match the requested last segment duration, not duration plus about 2 us.
  • Run debug.signal_timing and watch ov. A rising overrun count means the CPU missed a deadline.
  • If ov stays flat, any old inter-wave 2 us stretch should be removed by the continuous deadline schedule.
Limit of the fix: this does not make impossible schedules possible. If the pattern duration is shorter than the time needed for playback setup and required maintenance, the next wave will still be late. The difference is that the lateness is now reported as an overrun instead of being hidden as intentional hold time.
