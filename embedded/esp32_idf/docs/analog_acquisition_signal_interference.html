Why Core 0 Analog Acquisition Still Disturbs Core 1 Signal Timing

The diagnostic runs show that moving analog acquisition to Core 0 improves sample freshness, but oneshot ADC reads still perturb the timing-critical signal path. Different cores reduce direct task contention; they do not isolate shared peripherals, interrupt paths, memory/cache effects, or chip-level arbitration.

Unthrottled background acquisition 5555 triples/s Fresh samples, but the scoped 2 us dead-time degraded.
Rate-limited at 500 us 1519 triples/s Signal stability improved, but still worse than the older no-background-acquisition behavior.
Current conclusion Oneshot is noisy Rate limiting helps, but does not make the ADC producer hard-real-time safe.

Executive Answer

The signal loop is pinned to Core 1 and disables interrupts around GPIO playback, but the ESP32 is still one chip. A task on Core 0 that repeatedly calls adc_oneshot_read() can affect Core 1 through shared ADC/peripheral locks, APB and memory bus arbitration, cache/flash stalls, cross-core FreeRTOS/ESP-IDF critical sections, and system interrupts. The result is not necessarily visible as a firmware timing-fault counter, because the GPIO loop can still complete its average period while individual nanosecond-to-microsecond edge placement shifts enough to damage a scoped 2 us dead-time.

Timing Window

The signal code clears changing complementary outputs, waits the configured dead-time cycles, then writes the final GPIO state. That wait is only about 480 CPU cycles for a 2 us target at 240 MHz. Any disturbance that costs tens or hundreds of cycles is large enough to be visible on a scope.

Interpretation

The 500 us run is the most useful clue. It produces only about 1519 triples/s and still makes the scoped waveform worse than the old path. That means the disturbance is not only CPU saturation. The oneshot ADC call itself, or the driver/peripheral work around it, is likely introducing short stalls or arbitration delays that are significant relative to a 2 us dead-time.

The firmware counters are still useful, but they are not a substitute for the scope. Playback avg, Loop, and Overruns summarize software timing. The scope sees actual GPIO edge placement, including sub-loop perturbations and peripheral write timing.

Recommended Next Step

Option Expected effect Risk
Keep rate-limited oneshot Simple and measurable; can find a safe low rate. May never satisfy both 2 us dead-time and high-rate acquisition.
ADC continuous/DMA Moves conversion pacing into hardware/DMA, reducing driver-call frequency and per-sample CPU stalls. More implementation work; must handle DMA frames and overflow policy.
External ADC or dedicated sampling hardware Best isolation if internal ADC still disturbs timing. Hardware change and interface latency.

The current evidence says the next firmware milestone should be ADC continuous/DMA, not further pressure on oneshot. The acceptance test should remain the same: run the diagnostic, inspect the scope dead-time, and accept only rates that keep the 2 us blanking window stable while producing fresh enough analog triples.
