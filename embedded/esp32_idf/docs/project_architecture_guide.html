System Shape

The project has two running programs and one shared protocol. The ESP32 firmware owns the real-time side: GPIO signal generation, ADC sampling, BLE server, OTA receiver, and global device state. The browser dashboard owns the operator side: Web Bluetooth connection, controls, charts, logs, OTA upload, and local UI state.

Tutoring mental model: think of the browser as a lab console, protobuf as the cable protocol, and firmware as two machines: a communication/control machine on Core 0 and a timing-critical signal machine on Core 1.

Firmware Modules

Boot and task layout

app_main() is the firmware entry point. It initializes hardware, BLE, signal defaults, LED management, and then starts FreeRTOS tasks. The important design choice is core separation: communication and slower maintenance work stay on Core 0, while waveform playback runs on Core 1. 

esp_err_t app_init() {
    led_init();
    analog_init();
    signal_controller_init();
    nvs_flash_init();
    ble_controller_init();
    signal_set_alpha(0.5f);
    led_create_manager_task();
    sem_analog_read_trigger = xSemaphoreCreateBinary();
}

OTA Update Flow

OTA is browser-driven and BLE-transported. The dashboard reads a firmware binary, splits it into chunks, sends them as OtaCommand messages, waits for periodic firmware status acknowledgements, then sends OtaEnd. Firmware writes into the inactive OTA partition and reboots after setting it as the boot target.

Analog Data Acquisition

Between commits 697726a and 96bd2eb the analog subsystem was rewritten from a blocking oneshot reader into a non-blocking DMA-based producer/consumer pipeline. The changes follow the seven milestones defined in docs/plan_improve_data_aquisition.html.

Before vs After: control-cycle timeline

The old architecture called adc_oneshot_read() from Core 0 tasks, blocking the CPU for each sample. The new architecture offloads sampling to hardware DMA; Core 0 only parses short frames and publishes snapshots. Core 1 reads a pre-built snapshot — never an ADC driver.

DMA data flow

Seqlock snapshot publication

The producer writes triples through analog_publish_triple() which uses a seqlock pattern: increment s_snapshot_seq to an odd value (write in progress), store all six fields with relaxed ordering, then increment to even (write complete). Consumers in analog_get_status() and analog_read_control_snapshot() spin-retry if the sequence is odd or changed during the read. 

// producer — analog_publish_triple()
s_snapshot_seq.fetch_add(1);  // odd = writing
s_snapshot.raw_an3.store(raw_an3);
s_snapshot.raw_an5.store(raw_an5);
s_snapshot.raw_an6.store(raw_an6);
s_snapshot.calibrated_an3.store(calibrated_an3);
// ... an5, an6, timestamp, valid
s_snapshot_seq.fetch_add(1);  // even = complete

// consumer — analog_get_status()
for (;;) {
    seq_start = s_snapshot_seq.load();
    if (seq_start & 1) continue;  // write in progress
    // read all fields...
    seq_end = s_snapshot_seq.load();
    if (seq_start == seq_end) break;  // consistent
}

Per-channel queue parser

DMA frames deliver samples in hardware-determined order. The parser routes each sample into one of three ring queues (AnalogTripleAccumulator, depth 64). When all three queues have at least one sample, the oldest from each is popped, calibrated through the LUT, and published as a complete triple. 

struct AnalogTripleAccumulator {
    uint32_t raw[3][64];
    uint8_t  head[3], tail[3], count[3];
};

// on each DMA sample:
analog_queue_push(acc, channel_index, raw);
analog_publish_queued_triples(acc); // pops when all 3 ready

Control consumption

The signal loop on Core 1 no longer reads raw ADC globals. Instead it calls analog_read_control_snapshot() which validates:

  • Snapshot seq has advanced since last read
  • Sample age is below CONTROL_MAX_ANALOG_AGE_US (280 us)
  • Calibration LUT is ready
  • Snapshot is valid (complete triple)

On failure, a miss is recorded. After 3 consecutive misses, control is automatically disabled and the system drops to open-loop signal generation.

DMA pacing and watchdog

The continuous step processes frames in a tight loop but yields to the scheduler every ADC_CONTINUOUS_FRAMES_PER_IDLE_DELAY (32) frames with vTaskDelay(1), keeping the Core 0 idle task alive for the watchdog timer. Between delays, taskYIELD() gives other Core 0 tasks a chance to run.

Digital Signal Generation

The signal subsystem converts high-level pattern data into a tight GPIO loop. The slow part parses and precomputes. The fast part uses only precomputed masks, cycle deadlines, and direct GPIO register writes.

Core data structures

struct SignalStep {
    uint32_t set_mask;        // GPIO bits to set high
    uint32_t clr_mask;        // GPIO bits to clear low
    uint32_t clear_mask;      // both complementary pins off during dead time
    uint32_t dead_time;       // compensated CPU cycles
    uint32_t duration_us;
    uint32_t duration_cycles;
};

struct DataSet {
    uint32_t time_durations[MAX_SIGNAL_SIZE];
    uint32_t modes_d4[MAX_SIGNAL_SIZE];
    uint32_t modes_d5[MAX_SIGNAL_SIZE];
    uint32_t modes_d6[MAX_SIGNAL_SIZE];
    SignalStep steps[MAX_SIGNAL_SIZE];
    MatrixData gain_k;
    float target[3];
    float alpha;
};

Signal loop sequence

signal_start_continuous() creates signal_loop_task pinned to Core 1. Each loop iteration first applies any pending dataset swap, then consumes fresh ADC globals for control correction if control is enabled, then runs execute_signal_pattern(). Short patterns are repeated in a batch before maintenance work so the task does not leave the hot path too often.

  1. Pick the active DataSet and apply control correction to segment durations when fresh ADC data exists.
  2. Enter the interrupt-disabled GPIO playback helper.
  3. For each segment, add its corrected duration to an absolute next_edge deadline.
  4. Clear changed complementary outputs, wait the segment dead-time cycles, then apply the final clr_mask and set_mask.
  5. Busy-wait until next_edge, counting overruns when the deadline is already missed.
  6. After playback, optionally trigger analog acquisition and update the timing snapshot for status/debug commands.

Alpha 0.4 example

For alpha = 0.4, the generated pattern has six segments: durations 75, 19, 75, 19, 75, 19 us and modes 4, 6, 2, 3, 1, 5. The total period is T = 282 us. Interpreting each decimal mode as d6 d5 d4 gives the digital distribution below.

Dead-time behavior

When a channel changes state, clear_mask first turns both complementary pins off; only after step.dead_time does the loop assert the new target pin. Unchanged channels still appear in the final clr_mask/set_mask, but they do not require the initial all-off window.

Dead time is computed per segment during precompute by comparing the new mode bits against the previous segment. Segment zero compares against the final segment, so the repeated waveform is protected at wraparound.

The user-facing command is signal.set_dead_time with time_us. The router converts microseconds to CPU cycles, subtracts measured overhead plus g_dead_time_tail_overhead_cycles, and recomputes both datasets.

For waveform diagrams, data selection, and acquisition sequencing, see signal_loop_task_logic.html.

    const request_id = this.requestId++;
    const encoded = encodeUiCommand({

    packet.set(encoded, 1);

}

BLE Router

The BLE router is the firmware boundary between raw Web Bluetooth writes and project-level actions. It lives in ble_controller.cpp as ble_router(), but it deliberately delegates real work to the OTA controller and UI command router.

Write path

The GATT write event copies incoming bytes into s_cmd_queue. ble_command_task reads that queue, builds a minimal write object, and calls ble_router(). This offload keeps Nanopb decoding, JSON parsing, flash writes, and signal state changes out of the BLE stack callback.

Adding a UI command

ble_ui_command_router.cpp owns the command table. Dispatch parses the JSON payload once, finds the matching handler, and returns a small UiCommandResultData. 

static UiCommandResultData handle_signal_set_cycle_interval(const UiCommandContext& ctx) {
    uint32_t cycles;
    if (!json_get_u32(ctx.json, "cycles", &cycles)) {
        return invalid_arg("Expected numeric cycles");
    }
    g_cycle_nrun = cycles;
    return ok("Cycle interval updated");
}

BLE + Protobuf

BLE carries three kinds of binary packets. The first byte is a project prefix; the remaining bytes are protobuf encoded. This keeps OTA, command requests, and device notifications distinct while sharing one BLE characteristic.

PrefixDirectionMessagePurpose
0x01Firmware to webBlePacketTelemetry, status, logs, OTA status, command result.
0x02Web to firmwareOtaCommandBegin, chunk, end, or abort OTA.
0x03Web to firmwareUiCommandNamespaced JSON commands for normal operation.
 
message UiCommand {
    string name = 1;
    string json = 2;
    uint32 request_id = 3;
}

message BlePacket {
    oneof payload {
        Telemetry telemetry = 1;
        SystemStatus status = 2;
        LogMessage log = 3;
        OtaStatus ota_status = 4;
        UiCommandResult command_result = 5;
    }
}
