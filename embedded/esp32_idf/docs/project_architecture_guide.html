System Shape

The project has two running programs and one shared protocol. The ESP32 firmware owns the real-time side: GPIO signal generation, ADC sampling, BLE server, OTA receiver, and global device state. The browser dashboard owns the operator side: Web Bluetooth connection, controls, charts, logs, OTA upload, and local UI state.

Tutoring mental model: think of the browser as a lab console, protobuf as the cable protocol, and firmware as two machines: a communication/control machine on Core 0 and a timing-critical signal machine on Core 1.

Firmware Modules

Boot and task layout

app_main() is the firmware entry point. It initializes hardware, BLE, signal defaults, LED management, and then starts FreeRTOS tasks. The important design choice is core separation: communication and slower maintenance work stay on Core 0, while waveform playback runs on Core 1.

    analog_init();
    signal_controller_init();
    nvs_flash_init();
    ble_controller_init();
    signal_set_alpha(0.5f);
    led_create_manager_task();
    sem_analog_read_trigger = xSemaphoreCreateBinary();
}

BLE router responsibility

The BLE write callback does not do the heavy command work directly. Incoming bytes are copied into s_cmd_queue, then ble_command_task synthesizes a minimal write event and calls ble_router(). That keeps protobuf decoding, OTA handling, and command side effects out of the BLE stack callback path.

ble_router() is deliberately small: it inspects the first byte, decodes the remaining payload with Nanopb, and hands the decoded message to the owning subsystem. Prefix 0x02 becomes an OtaCommand for ota_controller_handle_command(). Prefix 0x03 becomes a UiCommand for ble_ui_command_dispatch(). The router then wraps success, validation failures, unknown commands, or decode errors into a BlePacket.command_result notification so the web request id can be resolved.

Signal Flow and Control Logic

The signal subsystem converts high-level pattern data into a tight GPIO loop. The slow part parses and precomputes. The fast part uses only precomputed masks, cycle deadlines, and direct GPIO register writes.

Core data structures

struct SignalStep {
    uint32_t set_mask;        // GPIO bits to set high
    uint32_t clr_mask;        // GPIO bits to clear low
    uint32_t clear_mask;      // both complementary pins off during dead time
    uint32_t dead_time;       // compensated CPU cycles
    uint32_t duration_us;
    uint32_t duration_cycles;
};

struct DataSet {
    uint32_t time_durations[MAX_SIGNAL_SIZE];
    uint32_t modes_d4[MAX_SIGNAL_SIZE];
    uint32_t modes_d5[MAX_SIGNAL_SIZE];
    uint32_t modes_d6[MAX_SIGNAL_SIZE];
    SignalStep steps[MAX_SIGNAL_SIZE];
    MatrixData gain_k;
    float target[3];
    float alpha;
};

Why double buffering matters

BLE commands can arrive while the signal is running. The firmware avoids tearing by writing the inactive dataset, precomputing all masks, and setting g_ds_update_pending. The signal loop swaps its ctx.dataset pointer only between pattern batches.

Signal loop sequence

signal_start_continuous() creates signal_loop_task pinned to Core 1. Each loop iteration first applies any pending dataset swap, then consumes fresh ADC globals for control correction if control is enabled, then runs execute_signal_pattern(). Short patterns are repeated in a batch before maintenance work so the task does not leave the hot path too often.

  1. Pick the active DataSet and apply control correction to segment durations when fresh ADC data exists.
  2. Enter the interrupt-disabled GPIO playback helper.
  3. For each segment, add its corrected duration to an absolute next_edge deadline.
  4. Clear changed complementary outputs, wait the segment dead-time cycles, then apply the final clr_mask and set_mask.
  5. Busy-wait until next_edge, counting overruns when the deadline is already missed.
  6. After playback, optionally trigger analog acquisition and update the timing snapshot for status/debug commands.

Pattern generation and selection

Pattern data enters through signal.set_alpha or signal.set_pattern. Alpha generation fills durations, mode bits, target values, and gain data; explicit pattern upload fills the same duration and mode arrays from the request. Both paths target the inactive dataset, run signal_precompute_steps(), and request a swap. The running loop never reads partially written pattern arrays.

Precompute translates the three dataset mode streams into direct GPIO masks. modes_d6 drives U1, modes_d5 drives U2, and modes_d4 drives U3. For each logical channel, bit 0 selects the LOW-side pin and bit 1 selects the HIGH-side pin. The hot loop therefore uses only masks and cycle counts, not pin-name branching or string parsing.

Alpha 0.4 example

For alpha = 0.4, the generated pattern has six segments: durations 75, 19, 75, 19, 75, 19 us and modes 4, 6, 2, 3, 1, 5. The total period is T = 282 us. Interpreting each decimal mode as d6 d5 d4 gives the digital distribution below.

Dead-time behavior

When a channel changes state, clear_mask first turns both complementary pins off; only after step.dead_time does the loop assert the new target pin. Unchanged channels still appear in the final clr_mask/set_mask, but they do not require the initial all-off window.

Dead time is computed per segment during precompute by comparing the new mode bits against the previous segment. Segment zero compares against the final segment, so the repeated waveform is protected at wraparound. If any changed channel rises in the target segment, the step uses g_dead_time_cycles_up; otherwise it uses g_dead_time_cycles_down. The current UI command stores the same compensated value in both directions.

The user-facing command is signal.set_dead_time with time_us. The router converts microseconds to CPU cycles with the current ticks-per-microsecond value, subtracts the measured helper_delay_cycles(0) overhead plus g_dead_time_tail_overhead_cycles, clamps tiny requests to zero, and then recomputes both datasets. The companion command signal.set_dead_time_tail_overhead adjusts the tail compensation in cycles and reapplies the last valid microsecond request.

During playback, a segment shorter than dead_time + MIN_ACTIVE_HOLD_CYCLES is clamped and counted as a timing fault. Dead-time windows shorten the active hold after a boundary; they do not move the scheduled absolute boundary. For waveform diagrams, data selection, and acquisition sequencing, see signal_loop_task_logic.html.

async sendCommand(name: string, payload = {}) {
    const request_id = this.requestId++;
    const encoded = encodeUiCommand({
        name,
        json: JSON.stringify(payload),
        request_id,
    });

    packet.set(encoded, 1);

}

BLE Router

The BLE router is the firmware boundary between raw Web Bluetooth writes and project-level actions. It lives in ble_controller.cpp as ble_router(), but it deliberately delegates real work to the OTA controller and UI command router. Its job is to protect the BLE callback path, choose the correct decoder, and normalize UI command responses.

Write path

The GATT write event copies incoming bytes into s_cmd_queue. ble_command_task reads that queue, builds a minimal write object, and calls ble_router(). This offload keeps Nanopb decoding, JSON parsing, flash writes, and signal state changes out of the BLE stack callback.

Prefix decisions

PrefixRouter actionResponse behavior
0x02Decode the bytes after the prefix as OtaCommand and call ota_controller_handle_command().OTA progress/status is emitted by the OTA controller as the transfer advances.
0x03Decode as UiCommand, dispatch by command name, then fill a BlePacket.command_result.The router immediately sends a 0x01 + BlePacket result notification.
Anything elseReject the write and log the unsupported binary prefix.No command result is emitted because there is no decoded request id to match.

UI command dispatch

ble_ui_command_router.cpp owns the command table. During initialization, ui_command_router_init() registers names such as system.get_status, signal.set_alpha, signal.set_pattern, signal.set_dead_time, analog.read_once, and debug.signal_timing. Dispatch parses the JSON payload once, finds the matching handler, and returns a small UiCommandResultData. 

0x03 + UiCommand
  name: "signal.set_dead_time"
  json: {"time_us": 2}
  request_id: 42

  decode UiCommand
  ble_ui_command_dispatch(command)
  ble_ui_command_fill_result_packet(...)
  ble_send_protobuf(0x01 + BlePacket.command_result)

Result shape

Every UI command response includes the original request id, the command name, an ok flag, a short machine-readable code, a human-readable message, and optional JSON. Decode failures use the same result envelope with code = "decode_error", so the web dashboard can handle success and failure through one notification path.

Adding a UI command

  1. Add a handler in ble_ui_command_router.cpp that accepts a UiCommandContext and returns UiCommandResultData.
  2. Parse only the JSON fields the command needs, using the local helper functions.
  3. Register the dotted command name in ui_command_router_init().
  4. Call the command from the dashboard through BleManager.sendCommand(); the prefix and request id are added there.

1. Add the handler

Handlers are plain functions in ble_ui_command_router.cpp. They receive the decoded command and parsed JSON object through UiCommandContext. Return invalid_arg() for bad payloads and ok() after the subsystem state change succeeds. The example below is the existing cycle-interval pattern; a new command should follow the same shape with its own name and subsystem call. 

static UiCommandResultData handle_signal_set_cycle_interval(const UiCommandContext& ctx) {
    uint32_t cycles;
    if (!json_get_u32(ctx.json, "cycles", &cycles)) {
        return invalid_arg("Expected numeric cycles");
    }
    if (cycles == 0) {
        return invalid_arg("cycles must be greater than zero");
    }

    g_cycle_nrun = cycles;
    ESP_LOGI(TAG, "Set cycle interval to %lu", cycles);
    return ok("Cycle interval updated");
}

2. Register the command name

Registration is the only step that connects a dotted command name to the handler. Add the new command in ui_command_router_init() near related commands so system.list_commands reports it with the rest of the command table. 

void ui_command_router_init(void) {
    s_command_count = 0;

    register_command("system.list_commands", handle_system_list_commands);
    register_command("system.get_status", handle_system_get_status);
    register_command("signal.start", handle_signal_start);
    register_command("signal.stop", handle_signal_stop);

    // Command name to handler binding.
    register_command("signal.set_cycle_interval", handle_signal_set_cycle_interval);
    // Other command registrations continue below.
}

3. Call it from the dashboard

The web side does not manually build the packet prefix. It calls BleManager.sendCommand(), which JSON-encodes the payload, assigns a request id, wraps it as UiCommand, and prefixes the packet with 0x03. 

await bleManager.sendCommand("signal.set_cycle_interval", {
    cycles: 10,
});

4. What the router returns

For a valid request, the browser receives a BlePacket.command_result notification similar to this. For invalid JSON or a missing cycles, the same envelope is used with ok = false and code = "invalid_argument". 

{
  "request_id": 42,
  "name": "signal.set_cycle_interval",
  "ok": true,
  "code": "ok",
  "message": "Cycle interval updated",
  "json": ""
}
Design rule: ble_router() should stay a routing boundary. Command-specific validation belongs in command handlers, OTA state belongs in the OTA controller, and schema changes belong in proto/messaging.proto.

BLE + Protobuf Integration

BLE carries three kinds of binary packets. The first byte is a project prefix; the remaining bytes are protobuf encoded. This keeps OTA, command requests, and device notifications distinct while sharing one BLE characteristic.

PrefixDirectionMessagePurpose
0x01Firmware to webBlePacketTelemetry, status, logs, OTA status, command result.
0x02Web to firmwareOtaCommandBegin, chunk, end, or abort OTA.
0x03Web to firmwareUiCommandNamespaced JSON commands for normal operation.

BLE router path

Web Bluetooth writes arrive at the GATT characteristic in ble_controller.cpp. The write callback copies the bytes into a fixed-size command queue, and ble_command_task later calls ble_router(). The router is intentionally a thin boundary: inspect prefix, decode protobuf, call the owning subsystem, and send a protobuf notification back when a result exists.

StepCode pathWhat it owns
Receive writeble_controller.cpp GATT write eventCopy bytes, acknowledge if required, avoid doing command work inside the BLE callback.
Offloadble_command_taskRead queued bytes and synthesize the minimal write object consumed by ble_router().
Routeble_router()Dispatch by prefix: 0x02 to OTA, 0x03 to UI commands.
Handleble_ui_command_router.cpp or ble_ota_controller.cppRun the subsystem side effect and produce status/progress/result data.
Notifyble_send_protobuf()Encode BlePacket, prefix it with 0x01, and send it to the dashboard.

UI command contract

Normal dashboard actions use 0x03 + UiCommand. The command has a dotted name, JSON payload, and request_id. Firmware dispatches the name through the fixed table in ble_ui_command_router.cpp, so commands such as signal.start, signal.set_pattern, signal.set_dead_time, analog.read_once, and debug.signal_timing all return the same result shape.

Results travel back as 0x01 + BlePacket with the command_result oneof selected. The payload includes the original request_id, command name, boolean ok, short code, human-readable message, and optional JSON. The dashboard keeps a pending request map, then uses the matching request id to attach the result to the UI log/status stream.

OTA deliberately bypasses the UI command table. Prefix 0x02 decodes to OtaCommand and is sent directly to ota_controller_handle_command(), because chunk streaming has different throughput and progress semantics from normal command/response requests.

Protobuf schema file

proto/messaging.proto is the shared contract between firmware and the dashboard. It defines telemetry and status packets, OTA transfer messages, UI commands, and UI command results. The packet prefix is not part of the protobuf schema; it is a project-level byte added before the encoded message so the firmware can choose the correct decoder before parsing. 

message UiCommand {
    string name = 1;
    string json = 2;
    uint32 request_id = 3;
}

message BlePacket {
    oneof payload {
        Telemetry telemetry = 1;
        SystemStatus status = 2;
        LogMessage log = 3;
        OtaStatus ota_status = 4;
        UiCommandResult command_result = 5;
    }
}

The schema is generated twice: Nanopb C files for firmware (main/src/proto) and protobufjs JS/TS files for the web app (web/src/proto). Run task proto:gen after editing proto/messaging.proto.

OTA Update Flow

OTA is browser-driven and BLE-transported. The dashboard reads a firmware binary, splits it into chunks, sends them as OtaCommand messages, waits for periodic firmware status acknowledgements, then sends OtaEnd. Firmware writes into the inactive OTA partition and reboots after setting it as the boot target.

How to Start

The project is controlled mainly through task. The root Taskfile wraps ESP-IDF setup scripts and protobuf generation; the web Taskfile wraps Bun/Vite and the optional Go webserver build.

Firmware workflow

# list available root tasks
task

# build firmware
task build

# flash firmware to configured port
task flash

# open serial monitor
task monitor

# build, flash, then monitor
task build-flash-monitor

# use auto-detected serial port
task build-flash-monitor-auto

Web workflow

# install dashboard dependencies
task web:install

# run Vite dev server
task web:run

# build the dashboard
task web:build

Protobuf workflow

# after editing proto/messaging.proto
task proto:gen

# firmware only
task proto:gen:firmware

# web only
task proto:gen:web

First bring-up sequence

  1. Run task info and confirm the IDF path, tools path, and serial port.
  2. Run task build. If protobuf changed, run task proto:gen first.
  3. Flash with task flash or task flash-auto.
  4. Monitor boot logs with task monitor.
  5. Start the dashboard with task web:run, then open the Vite URL in Chrome or Edge.
  6. Connect through the dashboard. Use STATUS, then START, then inspect telemetry and timing.
Browser requirement: Web Bluetooth generally requires Chrome or Edge and a secure context. localhost is accepted for development.
