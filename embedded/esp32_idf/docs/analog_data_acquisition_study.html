Analog Data Acquisition

This document maps how the ESP32 firmware currently acquires AN3, AN5, and AN6, how those readings feed BLE telemetry and control correction, and what must change if acquisition should become a high-speed, measurable subsystem.

Executive Finding

The current design is good enough for low-rate monitoring and control feedback, but it is not yet a fast acquisition pipeline. The fastest safe direction is to first add precise counters, then split control acquisition from dashboard telemetry, then move the ADC path from sequential oneshot reads to continuous/DMA frames or a timer-driven raw-sample path.

There are three acquisition consumers today: periodic dashboard telemetry control-loop feedback manual read command. They all share the same per-port helper, so every path inherits the same cost profile: port switch, blocking ADC read, raw-to-voltage float math, linear calibration search, and error sentinel handling.

Current Flow

The periodic path is time-driven by vTaskDelay(pdMS_TO_TICKS(g_analog_monitor_period_ms)). The control path is pattern-cycle-driven by trigger_periodic_analog_read(). Both paths record only the three-read ADC duration into the same latency window.

Action To Code

1. Boot initializes ADC1 channels

Action: configure ADC1 once, using oneshot mode and 12 dB attenuation. Active acquisition later uses AN3, AN5, and AN6. 

void analog_init() {
    adc_oneshot_unit_init_cfg_t init_config1 = {
        .unit_id = ADC_UNIT_1,
        .clk_src = ADC_RTC_CLK_SRC_DEFAULT,
        .ulp_mode = ADC_ULP_MODE_DISABLE,
    };
    adc_oneshot_new_unit(&init_config1, &adc1_handle);

    adc_oneshot_chan_cfg_t config = {
        .atten = ADC_ATTEN_DB_12,
        .bitwidth = ADC_BITWIDTH_DEFAULT,
    };

    adc_oneshot_config_channel(adc1_handle, ADC_CHANNEL_4, &config); // AN3
    adc_oneshot_config_channel(adc1_handle, ADC_CHANNEL_6, &config); // AN5
    adc_oneshot_config_channel(adc1_handle, ADC_CHANNEL_0, &config); // AN6
}

2. One read performs channel selection, blocking conversion, float math, and calibration

Action: map the abstract port to an ADC channel, read one raw sample, convert it to volts, then run table interpolation. This helper is simple, but it is also the hot path. 

float analog_read_port(AnalogPort port) {
    adc_channel_t channel;
    switch (port) {
        case AnalogPort::AN3: channel = ADC_CHANNEL_4; break;
        case AnalogPort::AN5: channel = ADC_CHANNEL_6; break;
        case AnalogPort::AN6: channel = ADC_CHANNEL_0; break;
    }

    int raw_val = 0;
    if (adc_oneshot_read(adc1_handle, channel, &raw_val) == ESP_OK) {
        float voltage = ((float)raw_val / ADC_MAX) * VOLTAGE_MAX;
        voltage = esp32_calibration(voltage);
        return voltage;
    }
    return -99.0;
}

3. Periodic dashboard telemetry reads and sends one protobuf packet

Action: if BLE is connected, read the three channels, record the elapsed ADC time, and notify the dashboard. This path is useful for human visibility, but it should not be treated as a deterministic acquisition clock. 

for (;;) {
    if (ble_is_connected()) {
        uint32_t start = esp_cpu_get_cycle_count();
        an3 = analog_read_port(AnalogPort::AN3);
        an5 = analog_read_port(AnalogPort::AN5);
        an6 = analog_read_port(AnalogPort::AN6);
        uint32_t end = esp_cpu_get_cycle_count();
        analog_record_latency((end - start) / esp_rom_get_cpu_ticks_per_us());

        packet.which_payload = BlePacket_telemetry_tag;
        packet.payload.telemetry.an3 = an3;
        packet.payload.telemetry.an5 = an5;
        packet.payload.telemetry.an6 = an6;
        ble_send_protobuf(&packet);
    }
    vTaskDelay(pdMS_TO_TICKS(g_analog_monitor_period_ms));
}

Latency Budget

Hot cost Why it matters Current evidence Improvement direction
Sequential oneshot reads Three blocking conversions serialize acquisition time and add scheduler sensitivity. analog_reading_task and analog_action_task both call AN3, AN5, AN6 one after another. Use ADC continuous/DMA frames, or timer-triggered oneshot reads if DMA is too large a change.
Linear calibration search Every channel scans a 33-point table in the acquisition path. esp32_calibration() loops from index 0 to calib_numel - 1. Use ESP-IDF ADC calibration, binary search, fixed-point interpolation, or calibrate off the real-time path.
Per-sample BLE packet BLE notification and protobuf encode happen per dashboard sample. BlePacket.telemetry carries only one AN3/AN5/AN6 triple. Batch samples or downsample display telemetry while keeping local control samples fast.
FreeRTOS tick timing Dashboard telemetry period is not a precise acquisition clock. Periodic path uses vTaskDelay(pdMS_TO_TICKS(...)). Use a hardware timer or continuous ADC sample clock for measured acquisition.

Observability State

Observability is no longer empty. The system exposes ADC latency and BLE congestion, and the signal loop exposes a timing snapshot. The gap is that these metrics are not yet detailed enough to guide aggressive performance work without guessing.

Critical Gaps To Implement

  • Separate telemetry acquisition from control acquisition. Today both paths use the same helper and latency bucket. A faster system needs source-tagged metrics and independent rates.
  • Add sample sequence, source, and microsecond timestamp. Without those fields, dropped BLE packets and control sample age are invisible.
  • Count ADC read failures per channel. The -99.0 sentinel can enter telemetry/control math unless every consumer checks it.
  • Count BLE send outcomes. ble_send_protobuf() returns an error, but the analog tasks do not act on it; congestion can drop stale data without a visible drop total.
  • Restore a hard monitor-period floor or make telemetry intentionally lossy. The current command handler only rejects zero, so a UI command can request very high notification rates.
  • Expose latency histogram or percentiles. Min/max/avg is enough to see basic movement, but not enough to distinguish stable latency from rare jitter.

Fastest Possible Architecture

Recommended sequence

Phase Change Why first Verification
0 Add observability counters: source-tagged ADC latency, ADC failures, sample sequence, BLE send drops, and acquisition-to-control sample age. Prevents blind optimization. Status/debug command shows stable counters under normal and intentionally overloaded rates.
1 Make control sampling independent from dashboard telemetry. Keep control latest-sample updates local; downsample dashboard data. BLE display should not set the real-time acquisition ceiling. Control latency remains stable while dashboard monitor period changes.
2 Reduce hot-path math: raw sample first, calibration later, binary search or native ADC calibration, fixed-point where practical. Small change, measurable CPU savings. ADC latency average and p95 fall without changing measured voltage behavior outside tolerance.
3 Move to ADC continuous mode with DMA frames for AN3/AN5/AN6. This removes software-triggered blocking conversion as the core bottleneck. No frame overflow at target rate; sequence and timestamp prove consistent cadence.
4 Batch telemetry protobufs or send compact byte arrays for charts. BLE becomes a lossy visualization channel rather than a sample-by-sample bottleneck. Drop count stays bounded and UI remains responsive at the selected chart rate.

Plan to Improve

Design target: keep the signal loop deterministic, update calibrated analog values as fast as the ADC path can safely sustain, and keep BLE commands responsive by making telemetry explicitly lower priority than command replies. The control deadline is now stricter than "latest value": a fresh calibrated sample, control processing, and command generation must fit inside one signal cycle, currently fluctuating around 280 us.

Grill-me decisions already resolved from the current code and constraints

D1 · Signal loop isolation No ADC driver call, BLE send, protobuf encode, allocation, logging, or calibration scan belongs inside the interrupt-disabled signal playback block.
D2 · Background acquisition owner Analog acquisition should be owned by a dedicated acquisition path, preferably ADC continuous/DMA on ADC1 channels AN3, AN5, and AN6.
D3 · Calibrated publish contract A sample becomes visible to control and BLE only after raw values are validated and calibrated. Consumers must not see half-updated triples.
D4 · BLE command responsiveness BLE interaction is lower priority than signal generation, acquisition, and control. Replies may be delayed, but BLE work must not block the real-time path.
D5 · Observability before speed claims Performance work starts by counting acquisition latency, sample age, ADC failures, DMA overflow, BLE send failures, and telemetry drops.
D6 · Latest-value control model The control loop should consume a newly published complete calibrated snapshot each cycle and never perform a blocking ADC read.
D7 · 280 us cycle deadline Each control cycle needs a fresh calibrated sample, control calculation, and command update before the next signal command is emitted. This makes sequential oneshot reads a fallback only, not the target architecture.
D8 · Half-cycle freshness target At the control update point, the complete calibrated AN3/AN5/AN6 triple should be less than half a cycle old, measured with age_us.
D9 · Miss policy If the sample is stale or calibration misses the cycle, never block the signal loop. Reuse the previous correction, count the miss, and disable control after repeated misses.
D10 · Fast calibration table A precomputed raw-to-calibrated lookup table or fixed-point interpolation is acceptable and preferred over the current per-sample linear calibration scan.
D11 · On-demand BLE telemetry BLE telemetry should reply only by demand. User interaction does not need to be instant while control is running.
D12 · Next-cycle command application A computed command applies on the next cycle boundary. The firmware should never mutate timing mid-cycle.
D13 · Control disable threshold Disable closed-loop control after 3 consecutive control-deadline misses.
D14 · Sequential triple is acceptable AN3, AN5, and AN6 do not require true simultaneous sampling for the first implementation. A fast sequential ADC frame is acceptable.
D15 · Float-first control math Keep the control math in float for the first fast implementation, then move to fixed-point only if measurements prove float is the bottleneck.
D16 · First acquisition-rate target Target 4 complete calibrated AN3/AN5/AN6 triples per 280 us control cycle, about one complete triple every 70 us.
D17 · ADC overflow policy On DMA overflow, count the overflow, invalidate the partial frame, keep the last valid correction for that cycle, and let the 3-miss rule disable control if it repeats.
D18 · Calibration LUT placement Keep hot calibration lookup data in internal RAM for the acquisition path; const source tables may remain flash-backed when not on the hot path.
D19 · Core ownership Keep the signal loop on Core 1. Run acquisition/calibration on Core 0 or another non-interfering priority context, verified against BLE starvation.
D20 · Analog debug command Add debug.analog_timing exposing seq, age, rate, latency, misses, overflows, and the latest calibrated triple.
D21 · Control fault output behavior After the 3-miss threshold, disable closed-loop corrections but keep the open-loop signal pattern running and report the fault.
D22 · Acquisition starts at boot Start acquisition after ADC initialization so snapshots and timing counters are warm before control is enabled.
D23 · Snapshot value format Expose calibrated floats to the control loop first, while retaining raw ADC codes and optional fixed-point calibrated values for diagnostics.
D24 · On-demand telemetry history On-demand BLE telemetry returns the latest snapshot and counters by default. Add a ring buffer only if waveform inspection is required later.
D25 · Calibration accuracy target The fast LUT must preserve current calibrated voltage behavior within one ADC LSB equivalent or 5 mV, whichever is larger.
D26 · No silent slow fallback If ADC continuous/DMA cannot reach the D16 target, fail control enable with a structured error and keep open-loop signal generation available.
D27 · Stability acceptance window Prove the fast path with at least 60 seconds at target rate and zero signal timing regressions, zero unhandled overflows, and zero 3-miss trips.
D28 · Legacy monitor compatibility Keep the old analog monitor command name, but make it configure on-demand/debug behavior rather than periodic sending while control is active.
D29 · Periodic analog task disabled Remove or disable the existing periodic analog task when the fast acquisition engine is active; preserve compatibility only at the BLE command layer.
D30 · Analog acquisition module owns ADC state ADC driver handles, calibration LUTs, snapshot publishing, and analog counters belong behind one analog acquisition module.
D31 · Not-ready BLE response If BLE requests analog data before a valid sample exists, reply with structured not_ready plus counters and initialization state.
D32 · Observability-first migration Implement the snapshot and counters on the existing oneshot path first, verify behavior, then swap the producer to continuous/DMA.
D33 · Protobuf status first Analog debug/status data should be added to the protobuf status path immediately rather than starting as JSON-only command output.
D34 · Temporary ADC global wrappers Keep temporary wrappers for g_adc_an3, g_adc_an5, g_adc_an6, and g_adc_fresh during migration, then remove direct global access.
D35 · Debug-only forced read Keep a forced single-read calibration/debug command, but make it debug-only and unavailable while control is active.

Target data path

I1 Define the measurement contract

Add a compact snapshot type owned by the analog module: seq, timestamp_us, raw AN3/AN5/AN6, calibrated AN3/AN5/AN6, valid mask, source, and error flags. Publish it with a seqlock-style double read or a small critical section so consumers never observe mixed-channel data. Add age_us at control consumption time so the firmware can prove the sample met the 280 us deadline and the half-cycle freshness target from D8. The control-facing fields should be calibrated floats for compatibility with the current control math, with raw ADC codes retained for diagnostics.

Verification: command/debug output can show the latest complete snapshot and prove that seq increases monotonically. Implement this contract first on the existing oneshot producer, then preserve the same interface when the producer changes to ADC continuous/DMA.

I2 Add counters before changing the ADC driver

Count samples produced, complete triples published, ADC read failures by channel, calibration failures, min/average/p95/max publish latency, sample age at control consumption, DMA/frame overflow, telemetry batches sent, telemetry drops, BLE send failures, and command replies sent. Track deadline misses separately: stale sample, calibration late, control math late, and command update late. Track consecutive misses so D13 can disable control after 3 misses before stale feedback silently drives the plant. Expose these through a nested protobuf AnalogStatus status/debug message as defined by D20, D33, D37, and D42. When the threshold trips, report a control fault while leaving the open-loop signal pattern running.

Verification: every on-demand system.get_status response includes AnalogStatus, and debug.analog_timing show the protobuf-backed counters while the current oneshot implementation still runs, and the legacy ADC latency fields mirror the new stats.

I3 Move acquisition off synchronous oneshot reads

Replace the control-triggered ADC read with a background acquisition engine. Preferred path: ADC continuous mode on ADC1 with DMA frames containing AN3, AN5, and AN6. Conservative fallback: a hardware-timer-driven acquisition task that performs reads outside the signal loop. The first target is 4 complete calibrated AN3/AN5/AN6 triples per 280 us cycle, about 14.3k triples/s or 42.9k channel samples/s. This gives the control loop freshness margin without waiting on the ADC. True simultaneous sampling is not required for the first version; publish the triple only after all three sequential channel readings are present and calibrated. Keep the signal loop pinned to Core 1; run acquisition/calibration on Core 0 or a priority context proven not to interfere with Core 1 timing. Start this acquisition path at boot after ADC initialization rather than waiting for control enable.

Constraint check: the signal loop should only read the latest snapshot and compute corrections; it should never call adc_oneshot_read(), wait on a semaphore, or block on frame availability. The old periodic analog task should be removed or disabled whenever the fast acquisition engine is active. Temporary compatibility wrappers may feed the current g_adc_an3, g_adc_an5, g_adc_an6, and g_adc_fresh globals during the migration.

I4 Calibrate before publish, but keep calibration cheap

The acquisition task should convert raw readings into calibrated values before setting the snapshot as fresh. Use a precomputed raw-code lookup table or fixed-point interpolation so each published sample is calibrated without the current linear search. Place the hot lookup data in internal RAM when it is accessed by the acquisition path. Keep calibrated floats available for current control math; optionally keep raw values for diagnostics. Keep ADC driver handles, LUT ownership, snapshot publishing, and counters inside the analog acquisition module.

Verification: a controlled voltage input produces the same calibrated values as the old path within tolerance, while the publish latency counter improves. The tolerance target is one ADC LSB equivalent or 5 mV, whichever is larger.

I5 Make BLE on-demand and lower priority

Do not send telemetry from the ADC callback or high-priority acquisition path. Remove periodic BLE telemetry as a control-era assumption: on-demand commands should copy the latest snapshot and counters, then send replies from a lower-priority BLE context. If BLE is congested or control load is high, delay or drop telemetry replies before stealing time from signal generation, acquisition, or control. Do not add a telemetry history ring buffer until waveform inspection requires it. The dashboard should render a compact analog status block after the protobuf fields are added. Add controls only where an action must be commanded to run or test the subsystem, so the user can interact with the hardware without adding automatic work to the real-time path.

Verification: while acquisition runs at the selected maximum rate, signal timing and control deadline counters remain clean. BLE requests still eventually receive replies when the system has budget.

I6 Change analog.read_once semantics

Make analog.read_once reply on demand with the latest calibrated snapshot instead of forcing a new blocking read. If a forced read is still needed for lab calibration, expose it as a slower explicit debug command so normal command handling stays fast. If no valid sample has been published yet, return a structured not_ready result with initialization state and counters. The forced read command must be unavailable while control is active.

I7 Ramp to the fastest stable rate

Start with a conservative ADC frame rate, then raise it while watching overflow, publish latency, BLE drops, command latency, and signal timing counters. The maximum accepted rate is the highest rate with no signal-loop timing regression, bounded command reply latency, no unhandled ADC frame overflow, and no 280 us control-deadline misses. Because the command applies on the next cycle boundary, verification should confirm there is no mid-cycle mutation of the active timing array. Inject or simulate overflow conditions and verify the partial frame is discarded, the overflow counter increments, and control follows the miss policy. The acceptance run is at least 60 seconds at target rate with zero signal timing regressions, zero unhandled overflows, and zero 3-miss trips. If hardware cannot reach the D16 target, control enable should fail with a structured error instead of silently running a slower feedback loop.
