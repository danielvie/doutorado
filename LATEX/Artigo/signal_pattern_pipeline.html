Signal Pattern Pipeline

The pipeline converts a browser command or alpha value into a precomputed ESP32 switching pattern. Core 0 prepares the inactive buffer; Core 1 swaps to it only at a pattern boundary, keeping the GPIO loop deterministic.

The inactive dataset is the handoff point: it absorbs command jitter before the real-time loop sees the new pattern.

The new waveform is ready before the boundary, but Core 1 does not change buffers until the current cycle finishes.

main/src/signal_controller.cpp:407 — Swap happens at the boundary

static void apply_pending_dataset_swap(SignalLoopContext &ctx) {
    if (!g_ds_update_pending.load(std::memory_order_acquire)) {
        return;
    }

    if (g_active_set.load(std::memory_order_relaxed) == SignalSet::SET_A) {
        ctx.dataset = &g_dataset_b;
        g_active_set.store(SignalSet::SET_B, std::memory_order_relaxed);
    } else {
        ctx.dataset = &g_dataset_a;
        g_active_set.store(SignalSet::SET_A, std::memory_order_relaxed);
    }

    g_ds_update_pending.store(false, std::memory_order_release);
    ctx.next_pattern_edge = 0;
    reset_control_corrections(ctx);
}

This is the vertical boundary in the timing chart. The pending flag can be set during a cycle, but this function is called by the signal loop before executing a complete pattern batch.

This is the Core 1 lane in the timing chart. The interrupt-disabled region is limited to waiting and GPIO register writes, so the browser command path cannot inject parsing or allocation into the hot loop.

Implementation constraint: never update g_dataset_a or g_dataset_b in place if that dataset is active. The whole pipeline depends on the active buffer being read-only until the next boundary swap.
